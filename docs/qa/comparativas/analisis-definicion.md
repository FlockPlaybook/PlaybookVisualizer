# 📊 Análisis y definición — Evolución por instancia

> Cómo crece esta responsabilidad a lo largo de las instancias.

---

## Instancia 1

Analiza flujos y funcionalidades con guía del referente. Identifica casos de prueba básicos. Solicita datos de prueba pertinentes y consulta ante requerimientos no comprende.

**Ejemplos:**
- Analizar requerimientos de baja complejidad con apoyo del referente.
- Identificar y documentar casos de prueba principales para features simples.
- Solicitar al equipo de desarrollo los datos de prueba necesarios.
- Hacer preguntas al referente cuando los requerimientos no son claros.
- Documentar los casos de prueba en la herramienta del proyecto (Jira, TestRail, etc.).

---

## Instancia 2

Analiza requerimientos e identifica flujos críticos de forma autónoma. Elabora documentación de casos de prueba completa. Prioriza y documenta casos adecuadamente.

**Ejemplos:**
- Analizar requerimientos de complejidad media e identificar flujos críticos.
- Elaborar documentación de casos de prueba de forma autónoma y completa.
- Priorizar casos de prueba según el riesgo e impacto de cada funcionalidad.
- Analizar feedback de usuarios finales para incorporarlo al plan de pruebas.
- Identificar dependencias entre casos de prueba y documentarlas.

---

## Instancia 3

Identifica, debate y fundamenta interrogantes de valor. Contempla casos borde del negocio en su totalidad. Empatiza con usuarios potenciales. Identifica posibles vulnerabilidades elementales.

**Ejemplos:**
- Cuestionar y debatir requerimientos con el equipo para identificar puntos ciegos.
- Contemplar casos borde, flujos alternativos y escenarios de error en el plan de pruebas.
- Empatizar con el usuario final para identificar comportamientos no previstos.
- Identificar posibles vulnerabilidades básicas de seguridad en las funcionalidades analizadas.
- Comprender la estructura interna de la API para diseñar pruebas más efectivas.

---

## Instancia 4

Define la estrategia de aseguramiento de calidad del proyecto. Colabora en estimaciones comerciales. Comprende en profundidad fundamentos de bases de datos. Detecta necesidad de pruebas no funcionales.

**Ejemplos:**
- Diseñar la estrategia de QA del proyecto (tipos de pruebas, cobertura, criterios de salida).
- Colaborar con el equipo comercial en estimaciones de esfuerzo de QA.
- Usar SQL para consultar bases de datos y validar la integridad de datos.
- Detectar la necesidad de pruebas no funcionales (performance, seguridad, accesibilidad).
- Definir criterios de aceptación de calidad para el proyecto.
